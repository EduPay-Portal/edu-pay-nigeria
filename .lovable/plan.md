\n# Wema Bank DVA Migration — Architecture & Refactor Plan\n\n## Goal\n\nPivot from Paystack-issued Dedicated Virtual Accounts (DVAs) to **direct Wema Bank Virtual NUBANs** as the primary collection method. Keep Paystack temporarily for **card top-ups only**. Build a **provider-agnostic payment layer** so future banks (Zenith, GTB, etc.) drop in cleanly. Reissue all DVAs via Wema once credentials arrive; archive existing Paystack DVAs as historical/inactive.\n\n---\n\n## Target Architecture\n\n```text\n                  ┌──────────────────────────────┐\n                  │   Frontend (React)           │\n                  │  - Admin DVA Mgmt            │\n                  │  - Reconciliation Panel      │\n                  │  - Top-up (card via Paystack)│\n                  └──────────┬───────────────────┘\n                             │\n                  ┌──────────▼───────────────────┐\n                  │  Edge Functions (Provider-   │\n                  │  Agnostic Service Layer)     │\n                  │                              │\n                  │  payments/                   │\n                  │   ├ providers/               │\n                  │   │   ├ wema.ts  (DVA)       │\n                  │   │   └ paystack.ts (cards)  │\n                  │   ├ dva-service.ts           │\n                  │   ├ webhook-router.ts        │\n                  │   └ reconciliation.ts        │\n                  └──────────┬───────────────────┘\n                             │\n                  ┌──────────▼───────────────────┐\n                  │  Postgres (Supabase)         │\n                  │  - virtual_accounts (multi-  │\n                  │    provider)                 │\n                  │  - transactions              │\n                  │  - webhook_events (unified)  │\n                  │  - reconciliation_logs       │\n                  │  - audit_logs                │\n                  └──────────────────────────────┘\n                             ▲\n                             │ webhooks (HMAC + IP allowlist)\n                  ┌──────────┴───────────────────┐\n                  │  Wema Bank API   │  Paystack │\n                  └──────────────────────────────┘\n```\n\n**Key principle:** every payment provider implements the same `PaymentProvider` interface. Business logic talks to the interface, never to a provider directly.\n\n---\n\n## Phase 1 — Provider Abstraction Layer\n\nCreate a shared module under `supabase/functions/_shared/payments/`:\n\n- `types.ts` — interfaces: `PaymentProvider`, `DVAProvider`, `WebhookEvent`, `NormalizedTransaction`, `DVAccount`.\n- `providers/wema.ts` — implements `DVAProvider` (auth, create NUBAN, verify txn, parse webhook). Stub now, live when creds arrive.\n- `providers/paystack.ts` — wraps existing Paystack code for **card top-ups only**. DVA methods deprecated.\n- `provider-registry.ts` — resolves provider by name (`wema`, `paystack`).\n- `secrets.ts` — central env var loader (`WEMA_CLIENT_ID`, `WEMA_CLIENT_SECRET`, `WEMA_API_KEY`, `WEMA_WEBHOOK_SECRET`, `WEMA_BASE_URL`, `WEMA_ENV`).\n\n`PaymentProvider` interface (sketch):\n```text\ncreateDVA(student) -> DVAccount\nverifyTransaction(reference) -> NormalizedTransaction\nparseWebhook(req) -> WebhookEvent | InvalidSignature\nlistSettlements(dateRange) -> Settlement[]\n```\n\n---\n\n## Phase 2 — Database Schema Changes\n\nA single migration adds provider-agnostic columns and new tables. Existing Paystack rows are preserved but marked inactive.\n\n**`virtual_accounts`** — extend:\n- `provider` already exists, default switches to `'wema'`.\n- Add: `provider_account_id text`, `provider_customer_id text`, `environment text` (`sandbox|production`), `status text` (`active|inactive|archived`), `assigned_at timestamptz`, `metadata jsonb`.\n- Backfill existing rows to `provider='paystack'`, `status='archived'`.\n\n**`transactions`** — extend:\n- Add: `provider text`, `provider_reference text` (replaces `paystack_reference` over time; keep old col for back-compat), `settlement_id uuid null`, `payer_account_name text`, `payer_account_number text`, `payer_bank text`, `match_status text` (`auto|manual|unmatched|over|under|duplicate`).\n\n**New: `webhook_events`** (replaces `paystack_webhook_events` for new events; old table retained read-only):\n- `provider`, `event_type`, `provider_reference`, `payload jsonb`, `signature_valid bool`, `ip_address inet`, `processed bool`, `processed_at`, `error_message`, `retry_count int`.\n\n**New: `reconciliation_logs`**:\n- `transaction_id`, `student_id`, `expected_amount`, `received_amount`, `match_type`, `notes`, `resolved_by`, `resolved_at`.\n\n**New: `settlements`**:\n- `provider`, `settlement_date`, `gross_amount`, `fees`, `net_amount`, `transaction_count`, `bank_reference`, `status`.\n\n**New: `audit_logs`**:\n- `actor_id`, `action`, `entity_type`, `entity_id`, `before jsonb`, `after jsonb`, `ip`, `created_at`.\n\n**New: `payment_provider_credentials`** (encrypted via pgsodium or stored only as env secrets — preferred env secrets for now, table reserved for multi-tenant future).\n\nAll new tables get RLS using `public.has_role(auth.uid(), 'admin')` per project memory.\n\n---\n\n## Phase 3 — Edge Functions Refactor\n\nRename / restructure:\n\n| Old | New | Purpose |\n|---|---|---|\n| `create-virtual-account` | `dva-create` | Provider-agnostic; defaults to `wema` |\n| `bulk-create-virtual-accounts` | `dva-bulk-create` | Sequential w/ backoff; provider param |\n| `paystack-webhook` | keep (cards only) | Strict scope: card charges → wallet top-ups |\n| _new_ | `wema-webhook` | HMAC + IP allowlist + timestamp; routes to reconciliation |\n| _new_ | `webhook-router` _(internal)_ | Normalises events from any provider into `webhook_events` |\n| _new_ | `reconcile-transactions` | Cron-callable; matches unmatched payments |\n| _new_ | `dva-reissue` | Bulk migrate students from Paystack DVA → Wema NUBAN |\n| `simulate-payment` | `simulate-payment` | Updated to simulate Wema webhook payloads too |\n\nWema webhook handler responsibilities:\n1. Verify HMAC signature using `WEMA_WEBHOOK_SECRET`.\n2. Validate request IP against `WEMA_ALLOWED_IPS` allowlist (configurable).\n3. Reject events older than N minutes (replay protection).\n4. Idempotency check on `provider_reference`.\n5. Insert `webhook_events`, then async-trigger reconciliation.\n6. Match payment → student via `virtual_accounts.account_number`.\n7. Create `transaction` (credit, completed) → existing trigger updates wallet.\n8. Detect over/under/duplicate → write `reconciliation_logs` + admin alert.\n\nAuto-VA creation trigger (`009_auto_virtual_account_creation.sql`) is updated to call `dva-create` with `provider='wema'`.\n\n---\n\n## Phase 4 — Frontend / Admin Dashboard\n\nRefactor and add:\n\n- `src/lib/payments/` — TypeScript mirror of provider types for the UI.\n- `VirtualAccountCard` — provider-aware (shows Wema branding when applicable).\n- `TopUpWalletDialog` — \
